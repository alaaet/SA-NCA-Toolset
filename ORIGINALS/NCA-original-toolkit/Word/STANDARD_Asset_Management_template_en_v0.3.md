# [Purpose](#_Purpose" \o "This section clarifies the importance and reasons for the development and adoption of this standard. This section also describes the standard's relationship with the requirements of NCA ECC, and organizational, legal, and regulatory requirements)

This standard aims to define the detailed cybersecurity requirements related to the asset management of <organization name>'s systems, data and information to minimize cybersecurity risks resulting from internal and external threats at <organization's name> in order to preserve confidentiality, integrity and availability.

The requirements in this standard are aligned with the cybersecurity requirements issued by the National Cybersecurity Authority (NCA) in addition to other related cybersecurity legal and regulatory requirements.

# [Scope](#_Scope" \o "This section of the standard template aims to identify the assets, parties and persons to which this standard applies)

This standard covers all assets (e.g., physical, data, business application, software and technology assets) in <organization name> and applies to all personnel (employees and contractors) in <organization name>.

# Standards

|  |  |
| --- | --- |
| 1 | Asset inventory |
| Objective | To create (and maintain) a secure repository of the information and technology assets owned and managed by <organization name>. The organization may create an asset inventory for each type of asset as defined in the Asset Management Policy |
| Risk implication | If <organization name> does not have or maintain an asset inventory, it will:   * Have an incomplete view of the assets owned and managed by the organization * Find it difficult to locate assets * Be unable to ensure that assets are up to date and being maintained as appropriate * Be unsure that it has the correct licensing or use arrangements in place * Be unable to protect all its assets from cybersecurity threats. |
| Requirements | |
|  | The asset inventory must be stored in a secure location. |
|  | The asset inventory must be protected by logical access controls (such as those defined in <organization name>’s Identity and Access Management Policy). |
|  | The asset inventory must be protected from unauthorized change by limiting access to only authorized individuals. |
|  | The asset inventory must be backed up on a regular basis and protected in accordance with the <organization name>’s backup and recovery policy and standard. |
|  | The asset inventory must be kept up to date, by adding assets once they have been owned or acquired and after they are deployed or disposed. |
|  | Details recorded in the asset inventory must be checked at least once a year for accuracy to ensure that the details are complete, comprehensive, correct and timely. |

|  |  |
| --- | --- |
| 2 | Asset inventory contents: critical and sensitive information assets |
| Objective | To record critical and sensitive information held by <organization name> |
| Risk implication | Having no or an incomplete inventory of critical and sensitive information will impair the ability of <organization name> to understand:   * What needs to be protected * Where it is stored * Which legal, regulatory and policy compliance obligations are associated with critical and sensitive information * How critical and sensitive information can be protected and handled. |
| Requirements | |
|  | For each critical and sensitive information asset (such as a merger and acquisition contract, salary details or marketing forecasts), the following information must be recorded in the asset inventory:   1. type of information being classified 2. level of classification of the information as defined in the <organization name> Data Classification Policy 3. date for reclassification 4. compliance requirements (e.g. recording whether asset is in scope of privacy, data retention or other statutory obligation) 5. systems, applications or processes that are dependent on the information for correct operation 6. identity of the information owner 7. asset location 8. asset owner (e.g. asset custodian) |

| 3 | | | Physical assets |
| Objective | | | To record the physical assets owned and managed by <organization name>. |
| Risk implication | | | Having no or an incomplete inventory of physical assets will impair the ability of <organization name> to understand:   * what needs to be protected * maintenance, licensing and protection requirements. |
| Requirements | | | |
|  | | | For each physical asset (network, IT, and specialist equipment), the following information must be collected and stored in the asset inventory:   1. type of asset (such as network, IT, and specialist equipment) 2. description of asset (such as firewall, server, Or PC) 3. manufacturer, make and model of asset 4. business purpose and/or business processes supported by the asset 5. owner (e.g. the individual who is accountable for the asset) and corresponding business unit 6. unique description or identifier (e.g. using serial numbers, network addresses or product numbers) 7. applications supported by the physical asset 8. physical location 9. level of criticality to <organization name> or classification given to asset 10. compliance requirements (e.g. asset is in scope of NCA or other regulator) 11. systems, applications or processes that are dependent on the physical asset for correct operation |
|  | | | The following information may be added to the asset inventory, though it is not mandatory:   1. associated hardware addresses (e.g. MAC address) 2. associated network addresses (e.g. IP address) 3. details of any software installed 4. details of any active ports, services or protocols on the device 5. whether hardware devices are approved for network connectivity 6. current connectivity status (e.g. is the equipment currently connected to corporate networks) 7. cybersecurity test results |
| 4 | | | Business applications and software |
| Objective | | | To record the business applications and software used by <organization name>. |
| Risk implication | | | Having no or an incomplete inventory of business applications and software assets will impair the ability of <organization name> to understand:   * what needs to be protected * maintenance, licensing and protection requirements. |
| Requirements | | | |
|  | | | For business applications, the following information must be recorded in the asset inventory:   1. business application name 2. application version number 3. application patch level 4. type of application such as customer relationship management (CRM) and collaboration platforms 5. business purpose and/or business processes supported by the asset 6. owner (e.g. the individual who is accountable for the asset) and corresponding business unit 7. information processed by each application, such as financial transaction data, sensitive business information or personally identifiable information 8. technical details about each application (e.g. supplier and licensing requirements) 9. cybersecurity test results 10. vendor support point of contact |
|  | | | For software, the following information must be recorded in the asset inventory:   1. name of software 2. software version number 3. software patch level 4. type of software (e.g. operating system, productivity software) 5. business purpose and/or business processes supported by the asset 6. owner (e.g. the individual who is accountable for the software) and corresponding business unit 7. technical details about the software (e.g. supplier and licensing requirements) 8. cybersecurity test results 9. vendor support point of contact |

| Objective | | | To record the third parties and suppliers that supply <organization name> with goods and services |
| Risk implication | | | Having no or an incomplete inventory of third parties and suppliers will impair the ability of <organization name> to understand:   * the goods and services provided * the information exchanged, shared, processed, transmitted or stored with third parties and suppliers * the information and physical assets that need to be protected at <organization name> and the third parties and suppliers. |

|  | | Each contract with a third party or supplier must have an individual entry and unique identifier | |

| Objective | To review the asset inventory on a regular basis and ensure it is up to date | | |
| Risk implication | If the asset inventory is out of date, then <organization name> will possess an incorrect overview of its assets, leading to missed upgrades, maintenance requirements, licensing requirements and cybersecurity status. | | |

| 7 | Secure disposal |
| Objective | To dispose of IT equipment and sensitive information in digital and physical formats in a secure manner. |
| Risk implication | Sensitive information, trade secrets, bespoke software and algorithms can be exposed when IT equipment, documentation and paper-based records such as reports, designs and analysis studies are incorrectly disposed. Such exposure may lead to regulatory fines, loss of reputation, commercial harm and loss of trust from governments, customers and individuals. |

|  |  |
| --- | --- |
| Requirements | |
|  | Data storage (including disk drives, USB drives and removable USB devices) must be removed from all IT assets prior to disposal. |
|  | Paper-based records must be securely destroyed by using a cross-cut shredder that meets Deutsches Institut für Normung (DIN) 66399 standard as P-3 or higher |
|  | Paper-based records marked as classified “Top Secret” and  “Secret” must be securely destroyed by using a cross-cut shredder that meets DIN 66399 standard as P-5 or P-6; or by incineration. |
|  | All identifying marks, such as asset tags, must be removed from any IT asset to be donated, sold or returned to a leasing organization. |
|  | All physical and hardware media assets that contained classified data must be destructed in a secure manner that ensures impossibility of data retrieve. |
|  | An approved information destruction supplier may be contracted by <organization name> to carry out the secure disposal process. |
|  | Certificates of destruction must be issued by the destruction supplier as a confirmation that the secure disposal process has been carried out. |
|  | The asset inventory must be updated with the relevant information about the disposal of the asset. |
|  | A disposal record must be created and include all necessary information about the disposal activities such as:   1. Date of disposal. 2. Asset disposed. 3. Type of asset. 4. Quantity. 5. Label or ID of asset. 6. Classification. 7. Who oversaw the disposal. 8. Disposal method. 9. Certificate of destruction if carried out by a supplier. |
